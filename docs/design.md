## Problem Statement

AI coding sessions already contain the engineering lab notebook developers rarely write by hand: timestamped intent, false starts, decisions, errors, and abandoned approaches. The raw artifacts (Claude Code JSONL at `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`, Codex JSONL at `~/.codex/sessions/`) are append-only, voluminous, and structurally rich, but they are not a journal. There is no way to ask "what did I try on `<project>` last month?" or "what dead ends did I already hit on this auth bug?" without scrolling thousands of lines of tool calls.

Jottrace turns those raw JSONL transcripts into local Markdown work-journal notes — and exposes a `recall` command so the journal becomes pre-work memory, not just post-work archive. The user maintained an earlier hermes-based work-journal collector at `~/.hermes/scripts/ai-work-journal/collector.py`; jottrace is a clean rebuild that drops the Hermes dependency, replaces cursor/hash state with a per-session SQLite ledger, and stops treating daily notes as the awkward main artifact. Hermes is not part of jottrace at any layer.

Source thesis: Simon Willison, "Coping strategies for the serial project hoarder" (2022): "the most useful engineering memory is often not polished documentation, but timestamped context about what was being attempted, what designs were considered, what failed, and what decisions were made." Jottrace operationalizes that thesis against a substrate Willison didn't have: AI coding agents that leave a complete machine-readable trail by default.

## What Makes This Cool

Two things, in this order:

1. **The negative-knowledge angle.** Most journaling tools optimize for "what worked." Jottrace explicitly preserves "what was tried, considered, and abandoned, and why." The JSONL has the false starts intact; existing renderers strip them. The user's word for this in conversation: *failure museum*. The sharper Codex-supplied frame: *negative knowledge engine* — the wins, dead ends, decisions, and abandoned branches that normal journals, git history, and agent logs all erase.

2. **Pre-work recall, not just post-work archive.** `jottrace recall <project>` runs before you start a new task and returns prior dead ends, wins, and decisions on that project. The journal is no longer something you write to; it is something you read from. This single behavior is what elevates jottrace above the existing landscape (claude-code-log: HTML rendering only; ccusage: spend analytics; manual journal habits: don't survive). It is also what justifies the SQLite topic-candidate ledger as MVP scope.

The user's stated goals, in order: "must work well for me," with OSS career signal as a secondary motivator. The negative-knowledge framing serves both: it is the differentiator that earns OSS attention, and it is the thing that makes future-Season actually open the journal on a Tuesday morning.

## Constraints

- **Personal-use first; OSS career signal second.** Decisions that improve daily-use ergonomics outrank decisions that improve commercial appeal.
- **Local-first.** No cloud sync, no telemetry, no required external account beyond the LLM provider the user already uses.
- **Markdown-first, viewer-agnostic.** Output is plain CommonMark with optional Obsidian-flavored extensions (frontmatter, callouts, internal links). Notes must read correctly in any Markdown viewer — Obsidian, VS Code, Marked, GitHub web, plain `cat`. No assumption that Obsidian is installed; no plugin required.
- **Output path is configurable.** Default is `~/jottrace-journal/` for users with no preference; users with an existing Markdown vault (Obsidian, Logseq, plain folder) point jottrace at that path — or, when the vault has hand-curated content, at a dedicated subfolder of it (recommended; avoids co-mingling jottrace-generated and hand-curated notes).
- **Append-only at the source layer.** Raw JSONL is never modified. Generated notes are regenerated when source revision changes.
- **macOS-first.** MVP ships and is tested on macOS arm64. Linux x64 is "should work, untested." Windows is out of scope.
- **No Hermes dependency.** Jottrace is standalone. The prior hermes-based collector is a legacy artifact; jottrace does not call, integrate with, or replace it via Hermes glue. Users migrating from the prior collector simply stop running it and start running `jottrace sync`.
- **Bun + TypeScript.** Distributed via `bun build --compile` static binary plus `bun add -g jottrace` from npm.
- **LLM is mandatory for content synthesis**, but the LLM never edits Markdown directly. Provider returns structured JSON (Zod-validated) only; a deterministic renderer writes the file. Deterministic skeleton is always written first; LLM synthesis fills sections asynchronously, so an offline run still produces a usable note.
- **LLM provider is pluggable, but MVP ships one.** A single `LLMProvider` interface keeps additional providers a one-file change post-MVP. MVP ships only `codex-cli` (shells out to the user's installed `codex` CLI; uses the existing Codex subscription; no API key required). Per-token API providers (`anthropic-api`, `openai-api`) are deferred indefinitely — flat-rate subscription holders are the target audience. Other future providers (`ollama`, `vertex`, etc.) are post-MVP exercises of implementing the interface.
- **Manual sync, no scheduler in MVP.** Users run `jottrace sync` themselves. The catch-up logic ("since last successful run") still applies regardless of when sync is invoked. launchd / systemd / cron sample files are post-MVP.

## Premises

Locked in this session. Each is load-bearing for downstream work.

1. **SQLite is authoritative** for source discovery, source signatures, session revisions, processed status, persisted LLM summaries, and topic candidates. Schema below.
2. **Adapter-per-source.** Each harness gets an adapter that translates its native transcript shape into one canonical `Session`. Adapters do NOT call LLMs; they parse, normalize, and emit the canonical model.
3. **LLM provider returns structured JSON only.** A deterministic Markdown renderer writes the file. Schema is Zod-validated; on validation failure, the orchestrator retries once with the validator error appended to the prompt; on second failure, the session note keeps its deterministic skeleton and is marked for retry. After 3 total attempts the session is marked `skipped` and is not reattempted automatically.
4. **YAML frontmatter is the canonical metadata layer**, and is renderer-owned (re-emitted on every sync; hand-edits to frontmatter are lost on the next sync). The H1 title is also renderer-owned. Body content lives inside HTML comment marker pairs (`<!-- jottrace:<name>:start v=1 -->` … `<!-- jottrace:<name>:end -->`); content outside any marker pair (including sub-headings, prose, and personal annotations the user adds between sections) is preserved verbatim. The full set of marker names is enumerated under "Generated Regions Update Algorithm".
5. **Naming: Jottrace.** Locked.
6. **Manual sync is the MVP.** No scheduler shipped in MVP. Users run `jottrace sync` from their terminal; if they want it on a timer they wire up launchd / systemd / cron themselves. The catch-up window ("since last successful run") works regardless of cadence.
7. **Standalone CLI.** Jottrace runs as a single CLI binary. No shell-out into other journaling tools, no integration glue with prior systems. Drop the user's prior hermes-based collector to retire it.
8. **First-run setup wizard.** When `jottrace` is invoked and no config exists at `~/.config/jottrace/config.toml`, the binary prompts the user through interactive setup: (a) which discovered harness adapters to enable, (b) the Markdown output path (default suggested), (c) confirms `codex-cli` is available on PATH and runs `codex --version` as a fail-fast health check (no provider radio in MVP — codex-cli is the only shipped provider), (d) optional provider config (codex binary path, model id). Setup writes config and exits; the user reruns `jottrace sync` to actually process sessions.
9. **MVP auto-discovers available harness sessions.** During setup, every implemented adapter probes its known on-disk location (Claude Code: `~/.claude/projects/`, Codex: `~/.codex/sessions/` and `~/.codex/archived_sessions/`). Adapters that find at least one session are presented as "include / exclude (default include)" toggles. Adapters that find zero sessions are not shown. The user's enable-list is persisted to config and re-checked on subsequent runs (`jottrace sync` honors the persisted include-list; `jottrace setup` re-runs discovery and updates).
10. **Topic candidates are tracked from day 1; topic notes are NOT auto-generated in MVP.** The candidate ledger is populated on every sync from two source families: (a) deterministic — `cwd`, `git_branch`, repo basename, frequent file paths in tool calls, repeated error strings, `outcome`-tagged labels; (b) LLM-derived — the validated `topics[]` field of `ExtractedSummary`, written with `source_kind='llm'` so `recall <project> <topic>` matches both deterministic and LLM-named topics. Materialised topic notes (Markdown files) are entirely manual in MVP. Auto-materialisation thresholds are post-MVP once the candidate distribution can be observed in the wild.
11. **Reading surface model.** Session notes are the canonical record (program may rewrite them when source revision changes). MVP also ships a deterministic daily rollup at `<output>/daily/YYYY-MM-DD.md` — flat hour-bucketed list of session-note links per day, no LLM call. Topic notes (materialised) and richer daily narratives remain post-MVP. The reading surfaces in MVP are: session notes, daily rollup, and the `recall` CLI.
12. **Distribution targets.** GH Actions CI (lint + type-check + test + binary build), npm publish (Bun-required package), README with two screenshots (recall output, session note). Single-binary releases for darwin-arm64 (primary, tested) and linux-x64 (best-effort, untested). darwin-x64 / linux-arm64 are post-MVP. mkdocs / docs site is post-MVP.
13. **Deterministic skeleton always written first; in-progress sessions are skipped until quiet.** `jottrace sync` runs in two distinct phases: (a) **discovery + skeleton** — for every new or updated source file *that has gone quiet* (defined as `(now - source_mtime) > [sync].quiet_minutes`, default 30), write the deterministic skeleton (frontmatter + Timeline + Source Trace) immediately, regardless of LLM availability. Files where the quiet check fails are not visited at all — no ledger row, no skeleton, no daily-rollup entry — until they go quiet on a later sync. (b) **synthesis** — for every session whose `synthesis_status` is `pending` or `retry`, call the LLM provider with an evidence packet, validate the response, and update the generated regions of the note. If the provider is unreachable in phase (b), the skeleton-only note remains valid and the session stays `pending`. A separate `jottrace synth` command runs phase (b) in isolation.
14. **Subagent sidechains are separate session notes** with a `parent_session_id` link in frontmatter. Claude Code's `parentUuid` chain produces sidechain JSONL files; each becomes its own canonical `Session`.
15. **Provider auth resolution.** For `codex-cli` (the MVP provider): no jottrace-side auth; jottrace shells out to `codex exec` and inherits the user's Codex auth from `~/.codex/auth.json`.
16. **Single-instance enforcement.** A `~/.config/jottrace/jottrace.lock` lockfile is acquired at the start of `sync` / `synth`. Format: two lines, `pid=<int>` and `started_at=<ISO 8601>`. Stale-lock check via `process.kill(pid, 0)`; on ESRCH/ENOENT the holder is dead and the lock is reclaimed. If alive, second invocation prints a clear message and exits 0.
17. **Coverage-window query.** On every `jottrace sync` start, the catch-up logic reads `MAX(started_at) FROM sync_runs WHERE outcome='success'` as `last_success`. Discovery scans only files with `mtime > last_success - 60s` (60-second clock-skew slop). On a fresh DB, the backfill window is `config.sync.backfill_days` (default 60).
18. **Note path scheme.** Session notes write to `<output_path>/sessions/YYYY/MM/<harness>/<short-id>.md` where `<short-id>` is the first 12 hex chars of `sha256(harness + ':' + source_session_id)`. This deterministically normalizes across harnesses regardless of native id format (UUIDs, ULIDs, KSUIDs, snowflakes). Daily rollups write to `<output_path>/daily/YYYY-MM-DD.md`. Collision detection: `sessions.short_id` has a unique index; on UNIQUE violation the renderer falls back to the full 64-hex digest for that single session and persists the full digest in `short_id`. In practice the 12-hex form will essentially never collide (~1-in-2^48 per pair) but the indexed check is cheap and deterministic.
19. **Subagent dedup in recall.** When `recall` returns a parent session and a subagent sidechain in the same result set, both are listed; the subagent is rendered as a nested bullet under the parent ("↳ subagent: …") instead of a top-level entry, with a link to the subagent's own session note.
20. **Idempotency contract.** "Re-running sync is idempotent" means *the content of generated regions does not change*. The renderer reads existing region content and short-circuits the atomic write when new content equals existing content. mtime-only churn from no-op writes is a bug. Test: run sync twice in a row on a stable corpus; the second run's `sync_runs` row records `synthesis_done=0` and zero file writes.
21. **Duration shorthand.** `recall --since` accepts `30d`, `7d`, `12h`, `45m`. Parser regex: `^(\d+)(d|h|m)$`. Not ISO 8601.

## SQLite Schema

```sql
-- MVP schema. Single file at <data_dir>/jottrace.db
-- where <data_dir> is ~/.local/share/jottrace on Linux, ~/Library/Application Support/jottrace on macOS.
-- PRAGMA journal_mode = WAL;
-- PRAGMA busy_timeout = 5000;
-- PRAGMA foreign_keys = ON;

CREATE TABLE sessions (
  harness            TEXT NOT NULL,                -- 'claude-code' | 'codex'
  source_session_id  TEXT NOT NULL,                -- harness-native session id
  parent_session_id  TEXT,                         -- non-null for CC subagent sidechains
  source_path        TEXT NOT NULL,                -- absolute path to JSONL file
  source_revision    TEXT NOT NULL,                -- sha256 of file contents at last successful processing
  source_size        INTEGER NOT NULL,             -- bytes; cheap pre-check before sha256
  source_mtime       INTEGER NOT NULL,             -- unix seconds; cheap pre-check before sha256
  byte_offset        INTEGER NOT NULL DEFAULT 0,   -- last fully-parsed byte; enables incremental parse on growing JSONL
  started_at         TEXT NOT NULL,                -- ISO 8601
  ended_at           TEXT NOT NULL,                -- ISO 8601 (last_event_at)
  cwd                TEXT,
  project            TEXT,                         -- last segment of cwd
  git_branch         TEXT,
  model              TEXT,                         -- harness-reported model id (e.g. 'claude-sonnet-4-6', 'gpt-5.5')
  event_count        INTEGER NOT NULL,
  short_id           TEXT NOT NULL,                -- 12-hex prefix of sha256(harness + ':' + source_session_id); falls back to full 64-hex on collision
  note_path          TEXT,                         -- relative to output_path
  synthesis_status   TEXT NOT NULL DEFAULT 'pending', -- 'pending' | 'done' | 'retry' | 'skipped'
  synthesis_error    TEXT,
  synthesis_attempts INTEGER NOT NULL DEFAULT 0,   -- bumped on every full attempt; transitions to 'skipped' at >= 3
  processed_at       TEXT NOT NULL,                -- last successful (skeleton or synth) write
  PRIMARY KEY (harness, source_session_id)
);
CREATE UNIQUE INDEX sessions_short_id ON sessions (short_id);
CREATE INDEX sessions_started ON sessions (started_at);
CREATE INDEX sessions_project ON sessions (project);
CREATE INDEX sessions_synthesis_status ON sessions (synthesis_status) WHERE synthesis_status != 'done';
CREATE INDEX sessions_parent ON sessions (parent_session_id) WHERE parent_session_id IS NOT NULL;

CREATE TABLE topic_candidates (
  harness            TEXT NOT NULL,
  source_session_id  TEXT NOT NULL,
  label              TEXT NOT NULL,                -- normalized (lowercased, kebab) candidate label
  source_kind        TEXT NOT NULL,                -- 'cwd' | 'git_branch' | 'repo' | 'file_path' | 'error' | 'outcome' | 'llm'
  weight             REAL NOT NULL DEFAULT 1.0,    -- repeated occurrences within a session bump weight
  PRIMARY KEY (harness, source_session_id, label, source_kind),
  FOREIGN KEY (harness, source_session_id) REFERENCES sessions (harness, source_session_id) ON DELETE CASCADE
);
-- INSERT OR REPLACE on (harness, source_session_id, label, source_kind);
-- weight is incremented when the same triple is re-seen during extraction in the same session.
-- 'llm' source_kind is written after successful synthesis from ExtractedSummary.topics[]; weight=1.0 per topic.
CREATE INDEX topic_candidates_label ON topic_candidates (label);

-- Persisted, validated LLM output — recall reads from here, never re-parses Markdown.
CREATE TABLE session_summaries (
  harness            TEXT NOT NULL,
  source_session_id  TEXT NOT NULL,
  schema_version     INTEGER NOT NULL DEFAULT 1,
  summary_json       TEXT NOT NULL,                -- the validated ExtractedSummary as JSON
  synthesized_at     TEXT NOT NULL,
  provider           TEXT NOT NULL,                -- 'codex-cli' in MVP; future providers stored as their own name
  model              TEXT NOT NULL,                -- the model id used (e.g. 'gpt-5.5')
  PRIMARY KEY (harness, source_session_id),
  FOREIGN KEY (harness, source_session_id) REFERENCES sessions (harness, source_session_id) ON DELETE CASCADE
);

CREATE TABLE sync_runs (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  started_at      TEXT NOT NULL,
  ended_at        TEXT,
  outcome         TEXT,                            -- 'success' | 'partial' | 'error'
  sessions_seen   INTEGER NOT NULL DEFAULT 0,
  sessions_new    INTEGER NOT NULL DEFAULT 0,
  sessions_updated INTEGER NOT NULL DEFAULT 0,
  synthesis_done  INTEGER NOT NULL DEFAULT 0,
  synthesis_failed INTEGER NOT NULL DEFAULT 0,
  error_message   TEXT
);
```

Cheap-then-expensive revision check on every sync: `(source_size, source_mtime)` is compared to the ledger row first; only if either differs is `sha256(file_contents)` recomputed. The byte-offset column enables incremental parse on growing JSONL — essential for large CC sessions that can reach 50–200 MB.

## Canonical Session / Event Model

The shape every adapter emits and every downstream module consumes. Lock this before anything else uses it.

```ts
// src/models.ts

export type Harness = "claude-code" | "codex";

export type EventKind =
  | "user"            // user message turn
  | "assistant"       // assistant message turn
  | "tool_call"       // assistant invoked a tool (always paired with a later tool_result)
  | "tool_result"     // result from a tool_call
  | "system"          // hooks, permission changes, internal markers
  | "error";          // tool errors, validation failures, cancellations

export interface Event {
  ts: string;                         // ISO 8601 with timezone
  kind: EventKind;
  text?: string;                      // user/assistant message body
  tool?: string;                      // tool name on tool_call/tool_result
  tool_call_id?: string;              // pairs tool_call with its tool_result
  tool_input?: unknown;               // adapter-normalized JSON input
  tool_output?: string;               // textual result; truncated by adapter to ≤ 4 KB
  exit_code?: number;
  error?: string;
  extra?: Record<string, unknown>;
}

export interface Session {
  harness: Harness;
  source_session_id: string;          // CC: file basename UUID; Codex: session_meta.payload.id
  parent_session_id: string | null;
  source_path: string;
  source_revision: string;            // sha256 hex of file contents at parse time
  source_size: number;
  source_mtime: number;
  byte_offset: number;
  started_at: string;
  ended_at: string;
  cwd: string | null;
  project: string | null;
  git_branch: string | null;
  model: string | null;               // harness-reported model id (descriptive; not the LLM provider used by jottrace synthesis)
  events: Event[];
}
```

Adapter normalization rules:
- A CC `user` event with role=user → `{ kind:'user', text }`. CC `assistant` events → `{ kind:'assistant', text }`. CC `assistant` events whose content includes `tool_use` blocks emit one `tool_call` event per block (sharing the message timestamp); the matching `user` event with `tool_result` content blocks emits one `tool_result` per block. CC `system` / `permission-mode` / `last-prompt` / `attachment` / `file-history-snapshot` events → `{ kind:'system' }` with `extra.subtype` preserved.
- Codex `response_item` with `payload.type='message'` → `user` or `assistant` per `payload.role`. `response_item` with `payload.type='function_call'` → `tool_call`; `payload.type='function_call_output'` → `tool_result`. `event_msg` and `turn_context` → `kind:'system'` with `extra.kind` preserved. `session_meta` is consumed for Session metadata and not emitted as an Event.
- Tool outputs longer than 4 KB are truncated by the adapter, with policy depending on result kind: **normal results** keep the last 4 KB; **errored results** (non-zero `exit_code` OR non-null `error` field) keep the first 1 KB plus the last 3 KB, because the actual error message often lives at the top of a long traceback while the tail is unrelated retry/cleanup noise. Elision uses the marker `[…N bytes elided…]`. The packet builder may further compress.

## ExtractedSummary Schema (Zod)

```ts
export const ExtractedSummary = z.object({
  title:        z.string().min(8).max(120),
  summary:      z.string().min(40).max(800),
  timeline:     z.array(z.object({
    time:       z.string().regex(/^\d{2}:\d{2}$/),
    text:       z.string().min(8).max(280),
  })).max(20),
  wins:         z.array(z.string().min(8).max(280)).max(8),
  dead_ends:    z.array(z.object({
    what:      z.string().min(8).max(280),
    why:       z.string().min(8).max(280),
  })).max(8),
  decisions:    z.array(z.object({
    decision:  z.string().min(8).max(280),
    rationale: z.string().max(280).optional(),
  })).max(8),
  outcome: z.object({
    status:       z.enum(["success", "partial", "blocked", "research", "failed"]),
    worked:       z.array(z.string()).max(8).default([]),
    did_not_work: z.array(z.string()).max(8).default([]),
    unclear:      z.array(z.string()).max(8).default([]),

  artifacts:    z.array(z.string()).max(20),
  followups:    z.array(z.string().min(8).max(280)).max(8),
```

## LLMProvider Interface

```ts
// src/llm/provider.ts
import type { ExtractedSummary } from "./schema";
import type { EvidencePacket } from "./packet";

export interface LLMProvider {
  readonly name: "codex-cli" | string;             // MVP ships codex-cli; future providers extend the union
  readonly model: string;                          // for session_summaries.model

  // Returns validated JSON or throws. The orchestrator handles retry policy, not the provider.
  synthesize(packet: EvidencePacket, opts: {
    system_prompt: string;
    schema_hint: string;                           // a stringified summary of the Zod schema for the prompt
  }): Promise<unknown>;                            // raw object, validated by orchestrator

  // Used by the setup wizard to fail-fast on misconfig before MVP processes anything.
  healthCheck(): Promise<{ ok: true } | { ok: false; reason: string }>;
}
```

One implementation ships in MVP. The interface keeps additional providers a one-file change post-MVP.

### `codex-cli` provider (MVP)

Spawns `codex exec` for each synthesis call. Uses the user's existing Codex subscription auth — no API key handling on the jottrace side. Detection: setup wizard runs `which codex && codex --version`. If missing, the wizard shows the install command (`brew install codex` or equivalent) and lets the user retry or pick a different provider.

Invocation contract:
1. Write the system prompt + evidence packet + schema hint to a temp file (avoid argv length limits and shell-escaping).
2. Spawn `codex exec --skip-git-repo-check -s read-only "$(cat $TMPFILE)" -C <output_path> < /dev/null` and capture stdout. Use a 5-minute per-call timeout.
3. Codex output mixes prose with code blocks; the prompt instructs the model to emit a single fenced ` ```json ` block containing the response. Parse out the first JSON code block; everything outside it is discarded.
4. If parse fails or the result doesn't validate against the Zod schema, throw — the orchestrator handles retry/retry-cap.

This approach means the user pays for synthesis through their existing Codex subscription, not a separate API bill. Tradeoffs vs. direct API: ~1-2 second subprocess spawn overhead per session (acceptable at MVP volumes; sync runs sequentially); structured output is parsed out of mixed prose (less reliable than native structured outputs, but the retry-with-validator-error fallback covers it).

Adding future providers (`ollama`, `vertex`, etc.) post-MVP is a matter of writing one file implementing `LLMProvider`. Per-token API providers (`anthropic-api`, `openai-api`) are deferred indefinitely — the audience for jottrace is flat-rate subscription holders, and a per-token bill on top of the journal layer is a poor fit.

## Evidence Packet (LLM input)

The packet is a compact JSON document handed to the LLM. Token budget: target **8 000 input tokens**, hard cap **16 000**. Strategy when the raw session exceeds budget:

1. Always include: session metadata (harness, started_at, cwd, git_branch, model), the first user prompt, the last assistant message, the outcome span (tail).
2. Include all tool-call summaries (one line each: tool name + truncated args + truncated result).
3. Include all error-bearing or non-zero-exit tool results in full.
4. Include all `Edit`/`Write`/`NotebookEdit` tool inputs in summarised form (file path + diff size).
5. Include all user messages in full; assistant messages summarised to first + last 200 chars per turn.
6. If still over budget after 1–5: drop assistant messages first, then summarise tool-call args, then drop non-error tool results.
7. If still over the hard cap after step 6: truncate the `tool_calls` list itself, keeping the most recent 50 and the most recent 50 error-bearing entries; emit a top-level `truncated: { dropped: K, kept_recent: N, kept_errors: M }` field. The LLM prompt instructs the model to lower `confidence` to `"low"` when this field is present.

```ts
export interface EvidencePacket {
  session: {
```

## Skeleton placeholder content

The deterministic skeleton, written in phase A before any LLM call, populates each generated region with explicit placeholder content rather than empty markers:

```md
<!-- jottrace:summary:start v=1 -->
_synthesis pending — run `jottrace synth` after restoring LLM connectivity_
<!-- jottrace:summary:end -->
```

Phase-A-only notes are self-explanatory in any Markdown viewer; the placeholder text also makes the idempotency contract trivial (the renderer compares existing region content to the new placeholder and short-circuits the write).

## Generated Regions Update Algorithm

Session notes contain a deterministic spine and a synthesised body delimited by HTML comment markers:

```md
<!-- jottrace:summary:start v=1 -->
...LLM-written content...
<!-- jottrace:summary:end -->
```

Update algorithm (`updateGeneratedRegions(notePath, regions: Map<name, content>)`):
1. Read file. If absent, write the full template with placeholder regions.
2. Locate each `<!-- jottrace:<name>:start ... -->` ... `<!-- jottrace:<name>:end -->` block via regex anchored on the marker pair.
3. If a marker pair is missing or malformed, abort the update; log a warning to `sync_runs.error_message`; do NOT silently rewrite.
4. Otherwise, replace the content between markers (preserving the markers) with the new content.
5. Content OUTSIDE marker pairs is preserved verbatim. Users can add personal notes around the generated blocks; jottrace will not touch them.
6. Before writing: read the existing file contents; if the proposed new full file equals the existing contents byte-for-byte, skip the write entirely (preserves mtime, preserves the idempotency contract). Otherwise write atomically: write to `<notePath>.tmp`, fsync, rename.

Marker version (`v=1`) lets future schema changes break compat cleanly; MVP only ever writes `v=1`.

**Marker names emitted by MVP:**

| Region              | Marker name              | Source                                   |
|---------------------|--------------------------|------------------------------------------|
| Source Trace        | `jottrace:source-trace`  | renderer (deterministic)                 |
| Summary             | `jottrace:summary`       | LLM (`ExtractedSummary.summary`)         |
| Timeline            | `jottrace:timeline`      | LLM (`ExtractedSummary.timeline`)        |
| Wins                | `jottrace:wins`          | LLM (`ExtractedSummary.wins`)            |
| Dead Ends           | `jottrace:dead-ends`     | LLM (`ExtractedSummary.dead_ends`)       |
| Decisions           | `jottrace:decisions`     | LLM (`ExtractedSummary.decisions`)       |
| Outcome             | `jottrace:outcome`       | LLM (`ExtractedSummary.outcome`)         |
| Artifacts           | `jottrace:artifacts`     | LLM (`ExtractedSummary.artifacts`)       |
| Follow-ups          | `jottrace:followups`     | LLM (`ExtractedSummary.followups`)       |

Frontmatter and the H1 title sit *outside* the marker pairs but are still renderer-owned (re-emitted on every sync). Everything else outside marker pairs — sub-headings the user adds, prose between sections, personal annotations — is preserved verbatim.

## Daily Rollup Template

Written to `<output_path>/daily/YYYY-MM-DD.md`. Pure deterministic — no LLM call. One file per `started_at`-day; sessions with `started_at` after midnight UTC-equivalent in the user's local timezone are bucketed by their start day, so a session 2026-05-01 23:50 → 2026-05-02 01:30 lands in `daily/2026-05-01.md` at the 23:00 bucket. Daily rollups for past days are regenerable on later syncs when previously-deferred (in-progress, then quiet, then synthesised) sessions close.

Concrete shape:

```md
---
type: jottrace-daily-rollup
date: 2026-05-01
session_count: 7
projects:
  - jottrace
  - agent-bappeda
  - admin-v3
generated_at: 2026-05-02T09:14:08+08:00
---

# 2026-05-01 — 7 sessions, 3 projects

## 23:00

- 23:50 · `claude-code` · `agent-bappeda` · [Tanstack rebuild — sidebar refactor](sessions/2026/05/claude-code/abc123def456.md) · partial

## 21:00

- 21:14 · `codex` · `agent-bappeda` · [Investigate eval drift](sessions/2026/05/codex/789012abcdef.md) · success
- 21:02 · `claude-code` · `jottrace` · [Adapter discovery wiring](sessions/2026/05/claude-code/cafe5678abcd.md) · success
```

Hour buckets `## HH:00` are listed newest-to-oldest top-down. Within a bucket, lines are ordered by `started_at` DESC. Each line: `HH:MM · ` + harness + ` · ` + project + ` · ` + `[<title>](<note_path>)` + ` · ` + outcome status. `<title>` falls back to `(synthesis pending)` when no synthesised summary exists yet (the deterministic skeleton phase ran but phase B hasn't completed for this session). Session links are relative to `<output_path>` root.

The rollup is wholly renderer-owned — no marker regions; the entire file is regenerated each time. Hand-edits to a daily rollup file will be lost. (If users want to annotate a day, they should do so on the linked session notes, where the marker discipline preserves out-of-band content.)

## First-Run Setup Wizard

Triggered automatically when `jottrace` is invoked without an existing config file. Also runnable explicitly via `jottrace setup` to reconfigure.

Flow:
1. Print a one-paragraph "Welcome to jottrace" intro that names the project's job in plain language (negative-knowledge engine for AI sessions).
2. **Discover sources.** Run every implemented adapter's `discover()` method (probes its on-disk location, returns count of sessions found). Adapters with zero hits are skipped silently. Adapters with ≥1 hit are presented:

   ```
   We found AI session logs from these tools:
     [x] Claude Code:  ~/.claude/projects/  (1247 sessions, 31 unique projects)
     [x] Codex:        ~/.codex/sessions/   (203 sessions, 14 unique projects)
   Press space to toggle, enter to confirm. (Default: include all)
   ```

3. **Output path.** "Where should jottrace write your work-journal Markdown notes?" Default: `~/jottrace-journal/`. Probe the path; if it exists and contains files matching `sessions/*/*/*/*.md`, ask "Existing notes detected; jottrace will only add/update notes, never delete. Continue? [Y/n]". If the path is inside an Obsidian vault (heuristic: `<path>/.obsidian/` exists somewhere ancestral), say so explicitly: "Detected as part of an Obsidian vault — jottrace's frontmatter and links will work natively." If the path is the *root* of an Obsidian vault and contains hand-curated content (any non-jottrace `.md` files at the root or in non-`sessions/` non-`daily/` subdirs), recommend a subfolder: "This vault has hand-curated notes; recommend pointing jottrace at `<path>/jottrace/` (or similar subfolder) so generated content stays separate from your own. Use vault root anyway? [y/N]".
4. **LLM provider health check.** MVP ships only the `codex-cli` provider. Probe `which codex && codex --version`. If missing, print install instructions (`brew install codex` or equivalent) and offer to retry. There is no provider radio in MVP — additional providers are deferred (per-token API providers indefinitely; `ollama` etc. post-MVP).

5. **Backfill window.** "How far back should the first sync look? [60d]" — written to `[sync].backfill_days`.

6. Write `~/.config/jottrace/config.toml` and `~/.config/jottrace/jottrace.lock` (ensure dir exists, set 0700 on dir, 0600 on config). Print:

   ```
   jottrace configured. Run:
     jottrace sync             # process all your AI sessions
     jottrace recall <project> # query prior dead-ends, wins, decisions
   ```

The wizard does not call any LLM, does not write any session notes — its only side effect is writing config + creating the SQLite DB schema.

## Approaches Considered

### Approach A: Slim MVP — single Bun package, manual CLI (RECOMMENDED)

Single Bun package; two adapters (Claude Code, Codex); SQLite ledger + topic candidates + session_summaries; LLM synthesis behind a pluggable `LLMProvider` interface with one MVP implementation (`codex-cli`); deterministic Markdown renderer; first-run setup wizard; `jottrace setup`, `jottrace sync`, `jottrace synth`, `jottrace recall` CLI commands; GH Actions CI; npm publish; static binary via `bun build --compile`.

- Effort: **15–20 working days** (the 14-18 prior estimate, adjusted for: -2 days from cutting scheduler, +3 days for second LLM provider + setup wizard + auto-discovery).
- Risk: Low. Every component has known prior art; the unknowns are LLM-prompt tuning quality and `codex exec` structured-output reliability.
- Pros:
  - Smallest path to "I'm using my own tool every day."
  - Single artifact to maintain, document, ship.
  - Setup wizard cuts the documentation burden — README becomes "install, run setup, run sync" instead of three pages of config syntax.
  - Pluggable provider design from day 1 makes adding `openai-api` / `ollama` post-MVP a single-file change.
- Cons:
  - README story is good but not dazzling — "reads logs, writes notes" with the recall-and-failure-museum hooks underneath.
  - No automatic scheduling means users must remember to sync (mitigated by the catch-up window — manual once a week works fine).
- Reuses: claude-code-log heuristics for CC parsing; Codex JSONL session_meta + response_item envelope for discovery; bun:sqlite for ledger; `commander` or `@clack/prompts` for CLI + setup wizard; `child_process.spawn` / `Bun.spawn` for the codex-cli provider.

### Approach B: Slim MVP + bundled MCP server

A's CLI plus a bundled MCP server in the same repo that exposes `recall` and `list_sessions` as MCP tools. Claude Code, Codex, and any MCP-capable client can call jottrace mid-conversation: "did I work on this before?"

- Effort: 20–28 working days (A + 5–8 for MCP)
- Risk: Low-Med
- Pros: Strongest README story (Claude Code calling `jottrace.recall` mid-session); MCP is hot in 2026; forces clean factoring of recall logic.
- Cons: +5-8 days; another protocol surface; per-client config friction.
- Reuses: A's reuses + `@modelcontextprotocol/sdk`.

### Approach C: Slim MVP + live-tail daemon mode

A plus a `jottrace watch` mode that runs in background, `fs.watch`-es the harness JSONL files, and writes notes seconds after sessions end.

- Effort: 20–30 working days (A + 5–10 for daemon mode)
- Risk: Med (`fs.watch` macOS edge cases)
- Pros: Magical UX; removes "forgot to sync" failure mode.
- Cons: Daemon-mode bugs eat into ship velocity; battery/memory cost.
- Reuses: A's reuses + Bun's native `fs.watch`.

## Recommended Approach

**Approach A.** Ship the slim MVP. The negative-knowledge framing, the recall command, the topic candidate ledger, the pluggable provider model, and the first-run setup wizard are the load-bearing differentiators; those all live in A. The MCP server (B) and the live-tail daemon (C) are good additive features but their absence does not block MVP from being the daily-use tool the user wants. Design A's recall logic as a pure library function so a post-MVP MCP server is a thin protocol wrapper.

### Pipeline contract — `jottrace sync`

Phase A — discovery + skeleton:
1. Acquire `jottrace.lock`. Begin a `sync_runs` row. Install a SIGINT handler that closes the lock file and writes `sync_runs.outcome='partial'` before exit.
2. Read enabled adapter list from config. For each enabled adapter: call `discover()`, returning `(source_path, current_size, current_mtime)` tuples for files modified since `last_success - 60s` AND where `(now - current_mtime) > [sync].quiet_minutes`. Files failing the quiet check are excluded entirely — no ledger row, no skeleton, no daily-rollup entry — until they go quiet on a later sync. Total files processed in phase A is capped by `[sync].max_per_run` (default 200); excess files are deferred to the next sync.
3. For each source file: read ledger row by `source_path`. If absent OR `(size, mtime)` differ from ledger: this is a NEW or UPDATED session. If `current_size < byte_offset`, treat as a JSONL rotation/truncation: reset `byte_offset=0`, force full re-parse, set `synthesis_status='pending'`. Otherwise compute `sha256(file_contents)`; if sha matches existing row, only update `(size, mtime)` and continue. Otherwise proceed.
4. Adapter parses the source (incrementally from `byte_offset` if updating, else full) into a canonical `Session`.
5. UPSERT ledger row: `source_revision`, `source_size`, `source_mtime`, `byte_offset`, `started_at`, `ended_at`, `cwd`, `project`, `git_branch`, `model`, `event_count`, `short_id`, `synthesis_status='pending'`, `synthesis_attempts=0`, `processed_at=NOW()`. Resolve `short_id` collisions on `UNIQUE` violation by retrying with the full 64-hex digest.
6. Topic-candidate extractor pulls labels from the parsed `Session` and UPSERTs into `topic_candidates` with deterministic `source_kind` values (`'cwd' | 'git_branch' | 'repo' | 'file_path' | 'error' | 'outcome'`).
7. Deterministic renderer writes/updates the session note skeleton via the generated-region algorithm. Note path: `<output_path>/sessions/YYYY/MM/<harness>/<short-id>.md`.
8. After phase A finishes, the deterministic renderer regenerates `<output_path>/daily/YYYY-MM-DD.md` for every distinct `started_at`-day touched by sessions added or updated in this run (idempotency contract still holds — no-op writes are short-circuited).

Phase B — synthesis:
9. SELECT rows from `sessions` WHERE `synthesis_status IN ('pending', 'retry')` AND `synthesis_attempts < 3`, ordered by `started_at ASC` (oldest first — predictable progress). Phase B has no `max_per_run` cap; concurrency is bounded by `[synth].max_concurrent` (default 1 in MVP — sequential).
10. For each: build `EvidencePacket` from session events (token-budget strategy above). Print one progress line per session: `✓ N/M <harness>/<short-id> — <duration>` to stdout (or one NDJSON event when `--json` is set).
11. Call `provider.synthesize(packet, …)` → unknown. Validate against `ExtractedSummary` Zod schema. Bump `synthesis_attempts`.
12. On validation failure: retry once with the validator error appended to the prompt. Bump `synthesis_attempts` again. On second failure within the SAME sync run: set `synthesis_status='retry'`, store `synthesis_error`, leave skeleton-only note as-is, continue with the next session. The next sync run picks `synthesis_status='retry'` rows back up — until `synthesis_attempts >= 3`, at which point the row transitions to `synthesis_status='skipped'`. Manual override: `jottrace synth --force <session_id>` resets attempts and reattempts.
13. On success: persist the validated `ExtractedSummary` JSON into `session_summaries` (UPSERT on `(harness, source_session_id)`) so future `recall` queries are pure SQL retrieval + JSON parse. Fold `ExtractedSummary.topics[]` into `topic_candidates` with `source_kind='llm'` and `weight=1.0` per topic (UPSERT). Then deterministic renderer updates the note's generated regions with the validated summary. Set `synthesis_status='done'`. Update `processed_at`.
14. Close `sync_runs` row (`outcome='success'` if all attempted sessions resolved, `'partial'` if SIGINT'd or any session ended in `'retry'`/`'skipped'`, `'error'` if phase A failed mid-run).
15. Release lock.

`jottrace synth`: runs phase B in isolation against pending/retry sessions. Useful when an offline phase A produced skeletons that need filling.

### `jottrace recall` contract

```
jottrace recall <project>
jottrace recall <project> <topic>
jottrace recall --since 30d --limit 10
jottrace recall <project> --json
```

- Resolves `<project>` against `sessions.project` (exact match then case-insensitive `LIKE %project%`).
- If `<topic>` given: filter to sessions whose `topic_candidates.label` matches (normalized — lowercased, hyphens for whitespace — case-insensitive `LIKE`). Both deterministic and LLM-derived (`source_kind='llm'`) candidates participate.
- `--since`: Go-style duration shorthand (`30d`, `7d`, `12h`, `45m`); regex `^(\d+)(d|h|m)$`; filters `started_at >= NOW() - duration`.
- `--limit`: max sessions to include (default 5, hard max 25).
- **Ordering.** Default order is `started_at DESC` (most recent first). When `<topic>` is given, primary order is `MAX(topic_candidates.weight)` for that label DESC, tiebreaker `started_at DESC` — strongest topic match first.
- **Truncation footer.** When the SQL query matches more sessions than `--limit`, the rendered output appends a footer line: `… and N more (raise --limit to see them)`. The JSON output exposes the same count as `truncated_count`.
- Data path: SQL retrieval (joins `sessions` × `topic_candidates` × `session_summaries`) followed by in-process JSON parse of `summary_json`. The TS recall function decodes each row's `summary_json` and extracts `wins[]`, `dead_ends[]`, top 3 `decisions[]` per session. No Markdown re-parsing.
- Output (default): Markdown to stdout. `--json`: same data as a JSON object for piping into other tools.

```md
# jottrace recall: <project> [<topic>]
generated: 2026-05-01T22:50:11+08:00 · sessions matched: 3

## 2026-04-29 · `<project>` · `<branch>`
- Dead end: tried wrapping the migration in a transaction → deadlocked under concurrent writes.
  Why: optimistic locking was already in place upstream.
- Win: switched to per-row update via `WHERE updated_at = ?`.
- Note: <output_path>/sessions/2026/04/codex/abc123def456.md
```

## Open Questions (informational only — none block MVP implementation)

1. **Codex `session_meta` cwd reliability.** Some old archived rollouts may have stale or absent cwd. Adapter falls back to `(no project)` and still produces a valid Session.
2. **`max_per_run` semantics.** Caps **total sessions touched in phase A** (skeleton writes + ledger updates) per sync run. Phase B has no separate cap — every pending/retry session in the ledger up to `synthesis_attempts < 3` is attempted (the 1-retry budget within a sync is per session, not per run).
3. **codex-cli structured-output reliability.** The current Codex CLI mixes prose with code blocks; jottrace parses the first JSON code block out. If a future Codex CLI ships native `--output-schema`, switch the codex-cli provider to use it.
